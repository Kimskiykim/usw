# План MVP: persistent state для coding agent

Статус: черновик для отдельного обсуждения. Документ фиксирует функциональные
требования и границы MVP, но намеренно не определяет окончательный набор
Markdown-файлов, их пути, шаблоны или machine-readable schema.

## 1. Исходная проблема

Coding agent теряет рабочий контекст при завершении чата, смене сессии или
LLM-провайдера. Git сохраняет код и историю изменений, но не отвечает на
вопросы:

- какую цель сейчас реализуют;
- где проходит scope задачи;
- какие решения уже приняты и почему;
- что выполнено, а что только планируется;
- какие проверки реально запускались;
- почему работа заблокирована;
- какое действие должно быть следующим.

Из-за этого новая сессия вынуждена повторно исследовать проект, перечитывать
историю чатов и заново принимать уже принятые решения.

## 2. Продуктовое обещание MVP

> Один разработчик может остановить работу в любой поддерживаемой coding-agent
> сессии, открыть новую сессию или сменить LLM-провайдера и безопасно продолжить
> работу без чтения предыдущего чата.

Новая сессия должна восстановить цель, границы, прогресс, решения, фактическую
проверку, blockers и следующее действие, а затем сверить сохранённое состояние
с реальным Git/worktree.

## 3. Граница MVP

В MVP предполагаются:

- один разработчик;
- одна рабочая копия проекта;
- один активный исполнитель в каждый момент времени;
- последовательная, а не параллельная работа агентов;
- возможность сменить чат, сессию, coding agent или LLM-провайдера;
- developer-local execution state;
- Git как canonical источник кода и истории изменений.

Рабочее ограничение до отдельного решения: одна активная задача на workspace.
Поддержка нескольких paused-задач остаётся открытым вопросом.

## 4. Не цели MVP

В первую версию не входят:

- несколько разработчиков;
- параллельные агенты;
- роли и передача ownership между ролями;
- locks и scope reservation;
- shared live collaboration index;
- review queue и review lifecycle;
- delivery handoff;
- SDD providers и template packs;
- artifact resolver;
- универсальный project-management или issue-tracking слой;
- сохранение полной истории сессий и рассуждений агента.

## 5. Основные принципы

### 5.1. State reconstruction, а не memory dump

Цель — сохранить минимальное состояние, необходимое для безопасного принятия
следующего решения. Полные диалоги, chain-of-thought и подробная история
сессий не сохраняются.

### 5.2. Git остаётся источником истины для кода

Persistent state не копирует diff, содержимое файлов и Git history. Он хранит
только смысловую интерпретацию, anchors и ссылки, необходимые для продолжения.

### 5.3. Факты отделяются от планов

Выполненные действия, запущенные проверки и принятые решения должны быть явно
отделены от предполагаемых следующих шагов.

### 5.4. Latest state не смешивается с историей

Execution state отражает текущее состояние работы. Повторные checkpoints не
должны бесконечно увеличивать объём контекста.

### 5.5. Temporary и durable knowledge имеют разный lifecycle

Текущий фокус и следующее действие исчезают после завершения задачи. Решения и
ограничения, важные для будущих изменений, должны пережить очистку active state.

### 5.6. Completion выводится из условий

Агент не должен объявлять работу завершённой только по ручной смене статуса.
Готовность определяется acceptance criteria, прогрессом, blockers и свежестью
verification.

### 5.7. Provider neutrality

Состояние не должно зависеть от формата внутренней памяти конкретной модели,
чтобы его одинаково могли использовать Codex, Qwen и другие coding agents.

## 6. Функциональные требования

### F-01. Discover active work

При запуске в workspace агент должен определить:

- существует ли активная работа;
- в каком она состоянии;
- можно ли её продолжить;
- где находится минимальный контекст для восстановления.

Если активной работы нет, агент должен явно начать новую, а не использовать
остатки завершённого состояния.

### F-02. Start work

При начале новой работы агент должен зафиксировать рабочий контракт:

- устойчивый идентификатор или краткое название;
- цель и ожидаемый результат;
- acceptance criteria;
- scope;
- non-goals;
- известные ограничения и предположения;
- исходную точку Git;
- первоначальный план.

### F-03. Maintain execution state

Во время работы агент должен поддерживать:

- текущий план;
- фактически выполненные пункты;
- оставшиеся пункты;
- текущий фокус;
- изменённые области;
- blockers и зависимости;
- одно конкретное следующее действие.

### F-04. Remember decisions

Агент должен сохранять решения, которые нельзя надёжно восстановить из кода:

- выбранный подход;
- причину выбора;
- отклонённые существенные альтернативы;
- ограничения решения;
- предположения, которые ещё нужно проверить.

Не каждое локальное рассуждение является durable decision. Перед сохранением
агент должен оценить, понадобится ли решение после следующего checkpoint или
после завершения текущей задачи.

### F-05. Remember verification

Для каждой проверки должны быть известны:

- точная команда или операция;
- фактический результат;
- состояние source, к которому относится результат;
- изменялся ли релевантный source после проверки;
- какие обязательные проверки ещё не запускались.

MVP может не иметь cryptographic receipts, но обязан отличать свежую проверку
от потенциально устаревшей и `not run` от `passed`.

### F-06. Create checkpoint

Перед паузой агент должен:

1. Сверить сохранённый state с Git/worktree.
2. Обновить план и фактический прогресс.
3. Зафиксировать новые существенные решения.
4. Записать реальные проверки и результаты.
5. Записать blockers, риски и неизвестные.
6. Оставить одно проверяемое следующее действие.
7. Сохранить только актуальное execution state.

Невалидный checkpoint не должен уничтожать последнее валидное состояние.

### F-07. Resume work

Новая сессия должна:

1. Восстановить рабочий контракт.
2. Восстановить план и прогресс.
3. Прочитать актуальные решения и ограничения.
4. Сравнить сохранённый source anchor с Git/worktree.
5. Проверить существование указанных ссылок и областей.
6. Определить свежесть verification.
7. Проверить, остаётся ли следующее действие применимым.
8. Сообщить о drift или конфликте фактов.
9. Продолжить с ближайшего безопасного действия.

Resume не должен требовать чтения предыдущего чата или полного журнала
checkpoint-событий.

### F-08. Represent blocked work

Для blocked-состояния должны быть известны:

- конкретная причина;
- что уже проверено или попробовано;
- какое решение, информация или внешнее действие требуется;
- что можно выполнять независимо от blocker;
- следующее действие после разблокировки.

### F-09. Reconcile with reality

Агент должен обнаруживать расхождения между persistent state и проектом:

- изменился Git HEAD;
- изменился working tree;
- указанный файл отсутствует;
- выполненная задача снова стала неактуальной;
- verification относится к предыдущему состоянию source;
- следующее действие уже выполнено или больше неприменимо.

Drift не должен автоматически исправляться предположениями. Агент сначала
показывает расхождение и только затем выбирает безопасное действие.

### F-10. Determine completion

Перед завершением агент должен проверить:

- выполнены ли acceptance criteria;
- отсутствуют ли незавершённые обязательные пункты;
- разрешены ли blockers;
- выполнена ли требуемая verification;
- остаётся ли verification свежей;
- соответствует ли working tree ожидаемому результату;
- какие риски остаются открытыми.

Если условия не выполнены, работа остаётся active или blocked.

### F-11. Retain durable knowledge

Перед очисткой active state агент должен классифицировать накопленное:

- временный execution progress — удалить;
- код и историю изменения — оставить Git;
- долгоживущие решения — сохранить как project knowledge;
- существенные ограничения и известные риски — сохранить;
- локальное следующее действие — удалить;
- verification — сохранить или сослаться на неё только при сохраняющейся
  применимости.

Точное место хранения durable knowledge определяется после согласования
функционального контракта.

### F-12. Repair state safely

Система должна позволять:

- обнаружить невалидное состояние;
- сохранить последнее валидное состояние при ошибке обновления;
- заменить повреждённое состояние валидным checkpoint;
- перейти в понятное пустое состояние после явного завершения;
- не выдавать повреждённый state за успешно восстановленный контекст.

## 7. Lifecycle работы

Предлагаемые live-состояния:

```text
idle
active
paused
blocked
```

`ready_to_complete` является вычисляемым условием, а не вручную назначаемым
статусом. `completed` является событием завершения и очистки live-state, а не
долгоживущим live-состоянием.

Основные переходы:

```text
idle -> active
active -> paused
active -> blocked
paused -> active
blocked -> active
active -> completed event -> idle
paused -> completed event -> idle
blocked -> completed event -> idle
```

Завершение из `paused` или `blocked` допустимо только после проверки условий
completion, а не только из-за ручного запроса очистить state.

## 8. Классы сохраняемого состояния

Это логические классы информации, а не решение о количестве файлов.

### Work contract

Цель, acceptance criteria, scope, non-goals и ограничения задачи. Живёт до
завершения работы; отдельные части могут стать durable project knowledge.

### Execution state

План, прогресс, текущий фокус, blockers и следующее действие. Mutable и
latest-state-only; очищается после завершения.

### Decision state

Существенные решения, причины и ограничения. Часть решений живёт только в
рамках задачи, часть должна быть повышена до durable knowledge.

### Verification state

Команды, результаты, source anchor и freshness. Может устаревать при изменении
source.

### Reconciliation state

Минимальные anchors, позволяющие сравнить checkpoint с текущим Git/worktree.
Точный способ вычисления anchors требует отдельного решения.

### Durable project knowledge

Решения и ограничения, которые должны быть доступны будущим задачам после
очистки active execution state.

## 9. Основные пользовательские сценарии

### Сценарий A. Новая задача

```text
нет active state
-> сформировать work contract
-> создать первоначальный план
-> зафиксировать source anchor
-> начать выполнение
```

### Сценарий B. Пауза и новая сессия

```text
выполнить ограниченный шаг
-> сверить Git/worktree
-> создать checkpoint
-> завершить сессию
-> открыть новую сессию
-> resume
-> продолжить с next action
```

### Сценарий C. Drift после checkpoint

```text
создать checkpoint
-> source изменяется вне текущей сессии
-> resume обнаруживает drift
-> verification помечается потенциально stale
-> агент пересчитывает безопасный next action
```

### Сценарий D. Blocker

```text
обнаружить blocker
-> записать причину и требуемое действие
-> сохранить независимые возможные шаги
-> pause
-> после изменения внешнего состояния выполнить resume
```

### Сценарий E. Завершение

```text
проверить acceptance
-> проверить прогресс и blockers
-> проверить freshness verification
-> выделить durable knowledge
-> очистить execution state
-> перейти в idle
```

## 10. Критерии приёмки MVP

### AC-01. Возобновление без истории чата

После checkpoint новая сессия восстанавливает цель, прогресс, решения,
verification, blockers и next action без доступа к предыдущему диалогу.

### AC-02. Обнаружение drift

Если Git или working tree изменился после checkpoint, resume явно сообщает о
расхождении и не считает сохранённую verification автоматически свежей.

### AC-03. Честная verification

Система различает `passed`, `failed`, `not run` и потенциально stale. Она не
преобразует планируемую проверку в выполненную.

### AC-04. Однозначное продолжение

В active state существует одно конкретное, проверяемое следующее действие либо
один конкретный запрос на разблокировку.

### AC-05. Completion gate

Работа не очищается как завершённая при невыполненных acceptance criteria,
обязательных проверках или unresolved blocker.

### AC-06. Durable knowledge переживает завершение

После очистки execution state существенные долгоживущие решения и ограничения
остаются доступными будущей сессии.

### AC-07. Ограниченный размер state

Многократные checkpoints заменяют актуальное состояние, а не накапливают
историю сессий.

### AC-08. Безопасное восстановление

Невалидный новый checkpoint не повреждает последнее валидное состояние, а
валидный checkpoint может исправить повреждённый current state.

### AC-09. Provider-neutral resume

Checkpoint, созданный одним поддерживаемым coding agent, может быть прочитан и
продолжен другим без provider-specific полей.

## 11. Нефункциональные требования

- состояние должно быть человекочитаемым;
- обязательные поля и переходы должны проверяться детерминированно;
- обновление current state должно быть атомарным;
- отсутствие optional context не должно симулироваться выдуманными данными;
- paths и ссылки должны проверяться безопасно;
- формат должен оставаться небольшим и переносимым;
- источник каждого факта должен быть понятен;
- повреждение state не должно повреждать project files;
- implementation не должна требовать отдельного сервиса для single-developer
  MVP;
- persistent state не должен зависеть от истории конкретного чата.

## 12. План определения артефактов

К выбору файлов и шаблонов следует перейти только после согласования разделов
1–11.

Последовательность следующего этапа:

1. Для каждого класса state определить canonical owner.
2. Определить, что выводится из Git, а что требует persistent записи.
3. Определить lifecycle: mutable, append-only или derived.
4. Определить scope: developer-local или project-shared.
5. Определить обязательность и cardinality.
6. Проверить, какие классы можно безопасно объединить.
7. Только после этого выбрать количество Markdown-файлов и их пути.
8. Согласовать минимальные шаблоны.
9. Определить validation и migration текущего прототипа.
10. Провести пилот на реальной задаче с полной сменой сессии.

## 13. Текущий прототип

Текущий flow `.usw/HANDOFF.md` и команд `usw-handoff`/`usw-resume` является
техническим spike для проверки checkpoint и resume. Его поля, статусы и
количество файлов не считаются утверждённым итоговым контрактом.

После согласования требований прототип следует:

- сопоставить с функциональными требованиями;
- сохранить полезные механизмы атомарной записи и validation;
- дополнить недостающими функциями;
- мигрировать или заменить без обязанности сохранять экспериментальный формат.

## 14. Будущее расширение до мультирольной команды

Мультирольный этап должен расширять, а не перепроектировать single-developer
core. Он потенциально добавит:

- несколько активных work items;
- identity, owner и role;
- передачу ответственности;
- shared active-work index;
- locks или другой scope coordination;
- зависимости между параллельными изменениями;
- sync discussions;
- независимый review;
- delivery handoff;
- разделение developer-local и shared team state.

Проектирование этого слоя начинается после проверки single-developer MVP и
появления реального сценария параллельной или мультирольной работы.

## 15. Открытые решения для обсуждения

1. Достаточно ли одной active-задачи на workspace или нужны несколько paused
   work items уже в MVP?
2. Какие acceptance criteria являются обязательными для любой задачи, а какие
   задаются самой задачей?
3. Какой Git/source anchor минимально достаточен для обнаружения drift?
4. Должен ли resume сразу продолжать выполнение или сначала показывать
   восстановленное состояние разработчику?
5. Когда checkpoint создаётся обязательно: только по команде, перед концом
   сессии или также после значимых этапов?
6. Какие решения считаются durable project knowledge после завершения задачи?
7. Где проходит граница между persistent state и внешним issue tracker?
8. Нужно ли MVP поддерживать последовательное переключение между разными
   coding agents как обязательный acceptance scenario?
9. Как обрабатывать незакоммиченные изменения, созданные после checkpoint?
10. Должна ли явная команда finish иметь право очистить state при невыполненном
    completion gate или только переводить работу в abandoned/cancelled?

## 16. Условие завершения requirements-фазы

Requirements-фаза завершена, когда:

- согласована граница single-developer MVP;
- согласованы обязательные пользовательские сценарии;
- согласованы функции F-01–F-12;
- согласован lifecycle состояний;
- согласованы completion и retention rules;
- на открытые вопросы раздела 15 получены решения;
- только после этого разрешено проектировать окончательные state artifacts и
  Markdown templates.
