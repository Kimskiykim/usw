# Экспериментальный структурированный контракт version-2

Использовать этот reference только после явного `--structured`/`-s` opt-in.
Ordinary Markdown является default и не использует этот contract.

## Форма

Создавать один prose-first документ. Обязательны цель, контракт и порядок
действий. Каждый верхнеуровневый пункт является действием или управляющим
блоком, имеет короткое постоянное имя в kebab-case и прямо описывает executor.
Номер нужен только для чтения; ссылки используют постоянные имена.

В MVP не создавать поля `После`, декларации `Пишет` и раздел полномочий записи.
Обычная последовательность задаётся порядком списка, остальные переходы —
только необходимыми управляющими маркерами.

Использовать маркеры только когда соответствующая конструкция нужна:

- `CALL` — вызов с типом `SKILL`, `SCRIPT`, `FLOW`, `SUBAGENT` или `HUMAN` и
  точной целью в backticks; тип `MODEL` не разрешать;
- `GATE` — конечный набор результатов решения;
- `IF`, `ELIF`, `ELSE` — полные взаимоисключающие пути после решения;
- `LOOP` — ограниченный возврат с условием завершения, положительным пределом
  попыток и явным исходом при исчерпании;
- `PARALLEL` — явно запрошенные или однозначно описанные одновременные
  независимые действия.

Остальной смысл оставлять обычным человеческим текстом. Не добавлять маркер
ради единообразия. Не создавать произвольные переходы или неограниченные
повторения.

Runner исполняет только канонические позиции ниже. Поэтому верхнеуровневое
действие записывать одной строкой `N. \`имя\` — <описание и executor>`, child
`PARALLEL` — строкой `   - \`имя\` — <executor>`, а nested subagent action —
вложенным нумерованным пунктом. `GATE` routes, поля `LOOP`, `Аргументы` и
`Действия субагента` сохранять с теми же labels и indentation, что в примере.
Другой prose разрешён только как документация и не должен нести скрытый
переход или executor.

Каждый вызов записывать как unquoted uppercase `CALL <TYPE>`, после которого
идёт точная цель в backticks:

```markdown
CALL SKILL `usw-plan-small-steps`
CALL SCRIPT `scripts/check_plan.py`
CALL FLOW `review-flow`
CALL SUBAGENT `reviewer`
CALL HUMAN `owner`
```

Для `SCRIPT` перечислять аргументы отдельными буквальными значениями. Keyword
распознавать только в своей позиции: `CALL SKILL` с последующей целью `call`
вызывает skill с именем `call`. Не запрещать имена, совпадающие с keyword без
учёта регистра.

После каждого `CALL SUBAGENT` создавать вложенный нумерованный блок действий.
Каждому вложенному действию давать глобально уникальное постоянное имя. Вызов
внутри блока принадлежит ближайшему enclosing subagent, а не родительскому
flow. Не создавать inline payload вроде `WITH SKILL` и не требовать отдельный
nested flow ради одного вызова.

Выводить из описания только очевидный порядок, проверку, ветку, ограниченный
возврат или независимую работу и показывать вывод в документе. Если условие,
предел, executor или управляющая связь неоднозначны, задать один вопрос.
Статически неочевидную параллельность оставить последовательной либо уточнить.

При редактировании сохранять постоянное имя каждого логически прежнего
действия, даже если номер, условие или описание изменились. Не
переиспользовать удалённое имя для другого действия.

Для каждого решения перечислять все результаты и продолжение каждого. Для
`PARALLEL` перечислять именованные дочерние действия; не утверждать фактическую
независимость будущего исполнения.

## Необязательный binding входов и результатов

Не требовать action-specific input map. Общая задача запуска доступна всему
flow. Только если пользователь явно просит experimental binding и actions
действительно требуют разные входы, документировать их отдельным prose-разделом
`Структурированный вход`: каждый внешний ключ должен точно совпадать с
постоянным именем action. Не добавлять executable-поля `Вход`, `Выход` или язык
выражений в `Порядок действий`.

Runner передаёт `HUMAN`, `SUBAGENT` и `FLOW` только input их action name, а
следующим typed actions — completed результаты предыдущих верхнеуровневых
boundaries. Для `PARALLEL` результаты children сохраняются под их постоянными
именами. Учитывать этот binding при описании consumer action, не добавляя
скрытых переходов.

## Канонический пример

```markdown
# Flow: prepare-and-review

Подготовить план, провести две независимые проверки и получить решение человека.

## Контракт

- Версия: `version-2`

## Порядок действий

1. `prepare-plan` — подготовить план: CALL SKILL `usw-plan-small-steps`.
2. `review-plan` — CALL HUMAN `reviewer`; GATE: выбрать `accepted` или `needs-work`.
   - IF `accepted`: продолжить к `independent-checks`.
   - ELIF `needs-work`: продолжить LOOP `revise-plan`.
   - ELSE: запросить один из объявленных вариантов.
3. `revise-plan` — LOOP не более 2 попыток, пока `review-plan` не вернёт `accepted`.
   - Каждая попытка дорабатывает план: CALL SKILL `usw-plan-small-steps`.
   - После попытки: снова `review-plan`.
   - При исчерпании: передать решение человеку.
4. `independent-checks` — PARALLEL:
   - `check-scope` — CALL SUBAGENT `scope-reviewer`.
     - Действия субагента:
       1. `analyze-scope` — CALL SKILL `usw-brainstorm-solutions`.
   - `check-safety` — CALL HUMAN `security-reviewer`.
```

## Статическая проверка и отчёт

Разрешить validator относительно основного `SKILL.md` как
`../usw-run-flow/scripts/run_flow.py` и проверить сохранённый документ:

- shared root: `python3 <validator> validate --experimental-structured <flows.root> <name>`;
- local root: `python3 <validator> validate --experimental-structured --local <project-root> <name>`.

До validator выполнить лёгкую статическую проверку:

- обязательные секции и точная версия присутствуют;
- постоянные имена уникальны, внутренние ссылки разрешаются;
- каждый `CALL` имеет разрешённый uppercase type и точную цель в backticks;
- `MODEL` не используется как type;
- каждый `CALL SUBAGENT` содержит непустой вложенный нумерованный блок, все его
  имена глобально уникальны, а вызовы принадлежат enclosing subagent;
- каждый `GATE` имеет конечные варианты и полные продолжения;
- каждый `LOOP` имеет условие, положительный предел и исход при исчерпании;
- skills имеют точные доступные имена;
- scripts — существующие исполняемые regular files внутри проекта, не
  symlinks, и получают отдельные буквальные аргументы;
- nested flows существуют в выбранном root и безопасно разрешаются;

Не запускать executor и не подтверждать branch feasibility или parallel
independence. Сообщить имя, origin, путь, версию, постоянные имена, управляющие
блоки, executors и результат validator. Сообщить точную команду запуска:
`$usw-run-flow --experimental-structured <name> <task>` либо
`$usw-run-flow --experimental-structured --local <name> <task>`.
