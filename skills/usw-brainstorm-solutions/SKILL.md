# Структурный брейншторм решений

Превращать открытый вопрос в ясный, пригодный для принятия решения ответ.
Сохранять цель и границы задачи пользователя.

## Правила работы

- Отвечать на языке пользователя и переводить заголовки шаблона на этот язык.
- По умолчанию использовать короткий формат. Переходить к полному формату,
  когда задача содержит существенные ограничения, несколько заинтересованных
  сторон, неоднозначную причину или важные компромиссы между решениями.
- Предлагать два-три действительно разных пути. Не добавлять третий путь, если
  это лишь небольшая вариация предыдущего.
- Показывать рекомендуемый путь первым.
- Не выдавать гипотезу за установленную причину. Разделять подтверждённое,
  предполагаемое и неизвестное, которое нужно проверить.
- Для каждого пути объяснять механизм: почему действие влияет на причину и
  какой результат ожидается.
- Не давать абстрактных советов без механизма, компромисса и условий выбора.
- Если существенной информации не хватает, задать один уточняющий вопрос. Если
  можно безопасно продолжить, явно перечислить допущения и дать ответ.
- Не начинать реализацию, если пользователь попросил только провести брейншторм.

## Capability contract

- Inputs: ограниченная проблема, контекст и уже принятые решения.
- Permitted writes: none; shared artifacts изменяет provider-aware capability.
- Outputs: разные solution paths, recommendation и один first step.
- Return point: вернуть результат вызывающему flow; не выбирать и не вызывать
  следующий skill.

## Короткий формат — основной для MVP

Использовать для простых и умеренно сложных вопросов. Масштабировать каждый
раздел до нескольких предложений, а не разворачивать полный шаблон без нужды.

```markdown
## Задача

Какой результат хотим получить и, если известно, как поймём, что задача решена.

## Проблема

Что конкретно мешает получить результат сейчас.

## Предполагаемая причина

Почему это происходит. Явно отметить, если причина подтверждена или пока
остаётся гипотезой.

## Пути решения

1. Рекомендуемый — суть, механизм, главное преимущество и компромисс.
2. Альтернативный — суть, механизм, главное преимущество и компромисс.

## Первый шаг

Одно конкретное и проверяемое действие. Если причина не подтверждена — способ
сначала проверить её.
```

## Полный формат — для сложных задач

Использовать последовательность:

`Задача → Контекст → Проблема → Причина → Пути решения → Рекомендация → Первый шаг`

### Задача

Описать требуемый результат и критерий, по которому будет понятно, что задача
решена.

### Контекст и ограничения

Указать факты, условия, ресурсы и ограничения, влияющие на решение. Пропустить
раздел, если контекст простой и не меняет выбор.

### Проблема

Описать наблюдаемое препятствие, которое мешает достичь результата сейчас. Не
подменять проблему предполагаемой причиной.

### Причина

Разделить:

- подтверждённые причины;
- предполагаемые причины;
- неизвестное, которое нужно проверить.

### Пути решения

Представить варианты в таком порядке:

```markdown
### 1. Рекомендуемый подход

- Что делаем:
- Почему это сработает:
- Преимущества:
- Компромиссы и риски:
- Когда выбирать:

### 2. Альтернативный подход

- Что делаем:
- Почему это сработает:
- Преимущества:
- Компромиссы и риски:
- Когда выбирать:
```

Добавить третий вариант только тогда, когда это иной механизм решения. При
необходимости сравнить пути компактной таблицей по критериям, важным для задачи:
эффект, усилия, скорость, риск или обратимость.

### Рекомендация

Выбрать путь с учётом цели и ограничений. Объяснить выбор причинной связью:

`Причина → действие → ожидаемый результат`.

Если данных для выбора недостаточно, дать правило выбора вместо ложной
уверенности.

### Первый шаг

Назвать одно конкретное и проверяемое действие. Если причина пока не
подтверждена, первым шагом предложить способ её проверить.
