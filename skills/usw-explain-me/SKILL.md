# Explain Me — переводчик предложений агента

Capability boundary: this skill reads the supplied material, writes no project
artifact, and returns only the explanation. It never selects execution scope or
invokes another skill. Return point: immediately after the explanation.

Переводить предложение агента в понятное объяснение, сохраняя смысл, границы
уверенности и важные решения. Отвечать на языке пользователя.

## Выбор уровня

Использовать явно названный пользователем уровень. Понимать как уровни и
синонимы:

| Уровень | Названия | Цель |
| --- | --- | --- |
| 0 | `хлебушек`, `ELI5`, `совсем просто` | Помочь понять суть без технического словаря. |
| 1 | `просто`, `новичок` | Дать бытовое объяснение с минимумом нужных терминов. |
| 2 | `технически`, `разработчик` | Показать компоненты, ход изменения и компромиссы. |
| 3 | `экспертно`, `глубоко` | Точно разобрать архитектуру, контракты и риски. |

Если уровень не указан, выбрать 0 при признаках низкой когнитивной нагрузки
(«я не соображаю», «коротко», «не технарь»); иначе выбрать 1. Сообщить
выбранный уровень одной короткой строкой, только если он был выбран автоматически.
Пользователь может в любой момент попросить: «на уровень выше/ниже».

## Общие правила

- Сначала назвать итог: что агент предлагает сделать и какую проблему это
  должно решить.
- Отделять факты из предложения от предположений и неизвестного. Не выдавать
  догадку агента за подтверждённую причину.
- Не подменять объяснение ревью: не искать новые альтернативы и не менять код,
  пока пользователь прямо не попросит оценить или выполнить работу.
- Переводить сокращения и жаргон при первом упоминании; на уровне 0 заменять
  их обычными словами. Не использовать снисходительный тон.
- Если в предложении есть выбор, явно сказать, что именно решает пользователь,
  каковы последствия каждого варианта и безопасен ли выбор для отката.
- Если исходного текста агента нет в текущем контексте, попросить прислать
  план, сообщение, дифф, ошибку или ссылку на него. Не выдумывать содержание.

## Формат по уровням

### Уровень 0 — «хлебушек»

Дать 3–6 коротких пунктов без предпосылок о знаниях пользователя:

```markdown
## По-человечески

<одна фраза: что хотят изменить и зачем>

- Сейчас: <простое описание текущей боли>.
- Предлагают: <что сделают простыми глаголами>.
- Получится: <видимый результат для пользователя или продукта>.
- Важно решить: <выбор пользователя, если есть>.
- Риск: <только если он реальный и важный>.

**В одной фразе:** <суть предложения>.
```

Использовать аналогию только если она делает механизм яснее; не добавлять её
ради украшения. Для незнакомого слова сразу дать расшифровку в скобках.

### Уровень 1 — простой

Объяснить блоками: «Что происходит», «Что предлагает агент», «Зачем это
нужно», «Что может пойти не так» и «Что нужно решить». Оставить только термины,
без которых нельзя точно объяснить решение, и определить каждый одним простым
предложением.

### Уровень 2 — технический

Объяснить цель, затронутые части системы или файлы, последовательность
изменений, сохранённые ограничения, проверку результата и компромиссы. Указать
связь `проблема → изменение → ожидаемый эффект`.

### Уровень 3 — экспертный

Добавить архитектурную мотивацию, интерфейсы и инварианты, поток данных или
управления, edge cases, обратимость, наблюдаемость и сравнение альтернатив,
если они есть в исходном предложении.

## Особые входные данные

- Для плана объяснять каждый шаг через видимый результат, а не через название
  технологии.
- Для диффа сначала объяснять поведение, которое изменится, затем — роль
  ключевых файлов. Не пересказывать строки кода подряд.
- Для ошибки объяснять: «что сломалось → почему агент так думает → что его
  предложение проверит или исправит».
- Для статуса отделять уже сделанное от следующего шага и от блокера.

## Финальная проверка

Перед ответом убедиться, что человек на выбранном уровне сможет ответить на
три вопроса: «что меняется?», «зачем?» и «нужно ли мне что-то решать?».
