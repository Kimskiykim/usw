## ADDED Requirements

### Requirement: Составление конечного обхода результата
`usw-create-flow` SHALL выражать одинаковую обработку заранее неизвестного числа
групп каноническим блоком `FOR EACH`. Блок MUST ссылаться только на именованный
конечный список в результате ранее объявленного действия и MUST указывать одно
дочернее действие для текущего элемента. Произвольный пользовательский Markdown
MUST NOT быть обязан использовать канонический маркер.

#### Scenario: Explorer создаёт группы контекста
- **WHEN** пользователь просит сначала выделить группы контекста, а затем одинаково обработать каждую группу
- **THEN** составитель создаёт `FOR EACH`, ссылающийся на список групп результата explorer, и не разворачивает неизвестные группы заранее

#### Scenario: Группы известны при составлении
- **WHEN** все элементы уже явно перечислены и не изменятся до запуска
- **THEN** составитель MAY создать обычные именованные действия вместо динамического обхода

### Requirement: Источник обхода имеет минимальный контракт
Источник `FOR EACH` MUST быть результатом завершённого действия и MUST содержать
конечный упорядоченный список. Каждый элемент SHALL иметь уникальное постоянное
имя и ссылку на относящийся к нему контекст. Вычисляемые выражения, вложенные
коллекции и изменение списка во время обхода MUST NOT поддерживаться.

#### Scenario: Валидный список результата
- **WHEN** предыдущее действие вернуло конечный список элементов с уникальными именами и ссылками
- **THEN** runner принимает список как источник обхода

#### Scenario: Источник отсутствует или неоднозначен
- **WHEN** результат не содержит объявленного списка, список не конечен, имена повторяются либо ссылка элемента отсутствует
- **THEN** runner останавливается до вызова дочернего исполнителя с наблюдаемой ошибкой контракта

### Requirement: Snapshot фиксируется до первой итерации
До первого дочернего действия runner SHALL сохранить в журнале запуска
неизменяемый snapshot постоянных имён, порядка и ссылок всех элементов. Resume
MUST использовать сохранённый snapshot и MUST NOT заново читать или толковать
текущий результат источника. Зафиксированный план запуска MUST оставаться
неизменным.

#### Scenario: Первый запуск обхода
- **WHEN** источник завершён и обход ещё не имеет snapshot
- **THEN** runner проверяет и сохраняет snapshot до обработки первого элемента

#### Scenario: Источник изменился после остановки
- **WHEN** внешний artifact источника отличается при resume
- **THEN** runner продолжает по сохранённым именам и ссылкам либо останавливается, если сохранённая ссылка больше не разрешается безопасно

### Requirement: Один вызов обрабатывает не более одного элемента
`FOR EACH` SHALL обходить snapshot последовательно в зафиксированном порядке.
Один вызов `usw-run-flow` MUST выполнять не более одного экземпляра дочернего
действия для текущего элемента, сохранять его нормализованный результат и
возвращать управление. Параллельный и вложенный обход MUST NOT выполняться.

#### Scenario: Обработан текущий элемент
- **WHEN** дочернее действие текущего элемента завершилось со статусом `completed`
- **THEN** runner сохраняет результат и делает следующий элемент доступным только для следующего вызова

#### Scenario: Пустой список
- **WHEN** сохранённый snapshot не содержит элементов
- **THEN** runner завершает блок со статусом `completed` без вызова дочернего исполнителя

#### Scenario: Элемент остановил выполнение
- **WHEN** дочернее действие вернуло `failed`, `blocked`, `decision_required` или нарушение полномочий
- **THEN** runner сохраняет результат на текущем элементе и не переходит к следующему

### Requirement: Прогресс обхода безопасно возобновляется
HANDOFF SHALL сохранять постоянное имя блока, удостоверение snapshot, текущий
элемент, номер попытки и результат каждого завершённого элемента. Resume SHALL
выбирать первый незавершённый элемент snapshot. Состояние `outcome_unknown`
MUST запрещать автоматический повтор текущего элемента и переход к следующему.

#### Scenario: Продолжение последовательного обхода
- **WHEN** часть элементов имеет подтверждённый статус `completed`
- **THEN** следующий вызов выбирает первый незавершённый элемент без повторной обработки завершённых элементов

#### Scenario: Результат текущего элемента неизвестен
- **WHEN** исполнитель мог произвести побочный эффект, но подтверждённый результат отсутствует
- **THEN** HANDOFF сохраняет `outcome_unknown`, а resume требует восстановления результата или явного разрешения повторить этот элемент

### Requirement: Полномочия проверяются для каждого элемента
Объявленные изменения дочернего действия MUST входить в полномочия родительского
flow и SHALL проверяться до каждого вызова. Фактические изменения каждого
экземпляра MUST проверяться после результата теми же правилами, что для обычного
действия.

#### Scenario: Полномочия достаточны
- **WHEN** изменения дочернего действия входят в полномочия flow
- **THEN** runner MAY вызвать исполнителя для текущего элемента

#### Scenario: Элемент требует лишних полномочий
- **WHEN** declared или фактические изменения экземпляра выходят за полномочия flow
- **THEN** runner останавливает обход и не запускает следующий элемент
