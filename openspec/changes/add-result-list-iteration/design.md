## Context

Change `add-markdown-flow-composition` вводит ручной контракт
`experimental-1`, зафиксированный план и журнал результатов. Однако число
групп, найденных исследовательским действием, известно только после его
выполнения. Такие группы нельзя заранее развернуть в отдельные действия, а
обычный `LOOP` описывает повторную попытку, не обход коллекции.

Этот change является зависимым backlog-элементом и реализуется только после
архивации `add-markdown-flow-composition`.

## Goals / Non-Goals

**Goals:**

- выразить последовательный обход конечного списка из результата завершённого
  действия;
- зафиксировать состав и порядок списка до обработки первого элемента;
- выполнять один элемент за вызов и безопасно продолжать с него через HANDOFF;
- сохранить prose-first характер ручного контракта.

**Non-Goals:**

- общий тип списков, переменные или язык выражений;
- изменение, фильтрация, сортировка или дополнение списка во время обхода;
- вложенные и параллельные обходы;
- новый parser, scheduler или runtime dependency.

## Decisions

### 1. Одна каноническая форма `FOR EACH`

Составленный flow MAY использовать блок вида `FOR EACH \`group\` IN
\`explore-context.groups\``. Левая часть является локальным читаемым именем, а
правая — только ссылкой `<stable-action-name>.<result-list-name>` на результат
ранее завершённого действия. Другие выражения не поддерживаются.

Альтернатива: моделировать обход через `LOOP`. Она отклонена, потому что скрывает
источник элементов и смешивает обход списка с повторной попыткой одного
действия.

### 2. Элементы имеют постоянные имена и ссылки

Источник возвращает конечный упорядоченный список элементов с уникальным
постоянным именем и ссылкой на контекст элемента. До первой итерации runner
проверяет список и сохраняет неизменяемый snapshot имён, порядка и ссылок в
журнале запуска. Сам зафиксированный план не изменяется.

Альтернатива: хранить только числовой индекс во внешнем artifact. Она отклонена,
потому что изменение artifact между вызовами могло бы незаметно подменить
текущий элемент.

### 3. Один элемент за один вызов

Каждый вызов исполняет не более одного экземпляра дочернего действия для
текущего элемента, записывает его нормализованный результат и возвращает
управление. Следующий вызов берёт следующий незавершённый элемент из snapshot.
Пустой snapshot завершает блок без вызова исполнителя.

Альтернатива: обработать весь список одним вызовом. Она отклонена, потому что
теряет наблюдаемые границы, безопасный resume и остановку между побочными
эффектами.

### 4. Последовательность является единственным режимом первой версии

`FOR EACH` обходит элементы в зафиксированном порядке. Если нужна параллельная
обработка, она остаётся отдельным будущим change и потребует правил полномочий,
лимитов и агрегации результатов. `PARALLEL` текущего ручного контракта не
применяется неявно ко всему списку.

## Risks / Trade-offs

- [Большой список раздувает HANDOFF] → сохранять только постоянные имена и
  ссылки на контекст, не копировать содержимое групп.
- [Источник вернул дубликаты или изменяемые ссылки] → отклонять список до первой
  итерации и не запускать дочерний executor.
- [Результат элемента неизвестен после возможной записи] → сохранять
  `outcome_unknown` и запрещать автоматический переход или повтор.
- [Последовательный обход медленнее параллельного] → принять это ограничение до
  появления измеримого сценария для отдельного параллельного change.

## Migration Plan

1. Завершить и архивировать `add-markdown-flow-composition`.
2. Добавить канонический `FOR EACH` в инструкции составления и исполнения.
3. Расширить существующий HANDOFF plan/journal минимальными полями snapshot и
   текущего элемента.
4. Добавить проверки пустого списка, resume, ошибки источника и
   `outcome_unknown` без изменения версии `1`.

Откат удаляет поддержку `FOR EACH`; существующие `experimental-1` и линейные
flow версии `1` остаются совместимыми.
