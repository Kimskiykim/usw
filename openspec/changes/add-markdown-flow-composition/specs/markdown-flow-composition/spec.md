## ADDED Requirements

### Requirement: Пользователь выбирает версию создаваемого flow
`usw-create-flow` SHALL создавать линейный контракт версии `1` и
структурированный контракт `version-2`. Флаги `-s` и `--structured` SHALL
сразу выбирать `version-2`. Если ни один флаг не передан, система MUST до
составления документа спросить пользователя, какую из двух версий создать.

#### Scenario: Передан короткий structured-флаг
- **WHEN** пользователь вызывает `usw-create-flow -s` с именем и описанием процесса
- **THEN** система составляет `version-2` без дополнительного вопроса о версии

#### Scenario: Передан длинный structured-флаг
- **WHEN** пользователь вызывает `usw-create-flow --structured` с именем и описанием процесса
- **THEN** система составляет тот же контракт `version-2`, что и для `-s`

#### Scenario: Флаг версии отсутствует
- **WHEN** пользователь вызывает `usw-create-flow` без `-s` и `--structured`
- **THEN** система спрашивает, создать версию `1` или `version-2`, и не записывает flow до ответа

### Requirement: Составление завершается без исполнения
`usw-create-flow` SHALL принимать обычное описание процесса, создавать из него
именованный project-owned Markdown-flow выбранной версии и завершаться после
проверки и записи документа. Система MUST NOT запускать созданный flow или
какое-либо описанное в нём действие.

#### Scenario: Создан линейный flow
- **WHEN** пользователь выбирает версию `1` и предоставляет допустимую линейную последовательность skills и scripts
- **THEN** система сохраняет совместимый flow версии `1` и возвращает его имя и расположение без исполнения

#### Scenario: Создан structured-flow
- **WHEN** пользователь выбирает `version-2` и описание прошло проверку
- **THEN** система сохраняет структурированный Markdown-документ, сообщает его имя, расположение и версию и не заявляет о поддержке текущим runner

### Requirement: Structured-flow остаётся человекочитаемым контрактом
Документ `version-2` SHALL содержать цель, версию контракта, порядок действий
и необходимые управляющие связи. Каждое верхнеуровневое
действие или управляющий блок MUST быть нумерованным пунктом с коротким
постоянным именем. Номер SHALL использоваться только для чтения, а связи MUST
использовать постоянные имена.

MVP `version-2` MUST NOT создавать поля `После`, декларации `Пишет` или раздел
полномочий записи. Последовательность верхнеуровневых действий SHALL задаваться
порядком списка, а остальные переходы — только необходимыми управляющими
маркерами.

Канонический документ SHALL использовать `CALL`, `GATE`, `IF`, `ELIF`, `ELSE`,
`LOOP` и `PARALLEL` только для присутствующих в процессе конструкций. Остальной
смысл SHALL оставаться обычным человеческим текстом.

#### Scenario: Нумерация изменилась при редактировании
- **WHEN** логически прежнее действие переместилось на другую позицию
- **THEN** его номер может измениться, но постоянное имя и ссылки на него сохраняются

#### Scenario: Добавлен поясняющий текст
- **WHEN** документ содержит ненумерованный абзац, заголовок или примечание
- **THEN** текст остаётся контекстом и не создаёт отдельное действие

#### Scenario: Конструкция процессу не нужна
- **WHEN** описание не содержит решения, повторения или параллельной работы
- **THEN** составитель не добавляет соответствующие управляющие маркеры ради единообразия

### Requirement: Очевидная структура может быть выведена из описания
Составитель MAY преобразовывать однозначно следующий из текста порядок,
проверку, ветку, ограниченный возврат или независимую работу в каноническую
структуру. Каждый такой вывод MUST быть видим в записанном Markdown. Если разные
толкования меняют поведение, исполнителя, управляющие связи или внешние
последствия, система MUST задать один необходимый вопрос до записи документа.

#### Scenario: Описание однозначно задаёт проверку
- **WHEN** пользователь требует после подготовки проверить результат и при замечаниях вернуть его на доработку
- **THEN** составитель отображает проверку, ветки и ограниченный возврат как явные структурные блоки

#### Scenario: Толкования меняют исполнителя
- **WHEN** описание не определяет, должен ли проверку выполнить skill или человек
- **THEN** система уточняет исполнителя до составления этого шага

### Requirement: Исполнитель указан прямо
Каждое действие `version-2` SHALL прямо описывать исполнителя. Канонический
вызов MUST содержать unquoted uppercase-оператор
`CALL`, ровно один тип `SKILL`, `SCRIPT`, `FLOW`, `SUBAGENT` или `HUMAN` и
точную цель в backticks. Тип `MODEL` MUST NOT создаваться. Скрипт MUST получать
отдельные буквальные аргументы без shell-строки.

#### Scenario: Действие использует skill
- **WHEN** составитель связывает действие с доступным skill
- **THEN** документ записывает `CALL SKILL` и точное имя skill в backticks

#### Scenario: Действие использует script
- **WHEN** составитель связывает действие с project-local script
- **THEN** документ записывает `CALL SCRIPT`, безопасный project-relative путь в backticks и отдельные буквальные аргументы

#### Scenario: Имя совпадает с ключевым словом
- **WHEN** точное имя skill равно `call`, `skill` или другому зарезервированному слову без учёта регистра
- **THEN** uppercase keyword остаётся оператором или типом, а имя в backticks однозначно остаётся целью

#### Scenario: Запрошен model target
- **WHEN** описание предлагает непосредственный `CALL MODEL`
- **THEN** составитель не создаёт такой вызов и запрашивает допустимого исполнителя

### Requirement: Работа субагента имеет явную вложенную границу
Каждый `CALL SUBAGENT` SHALL содержать вложенный Markdown-блок нумерованных
действий. Каждое вложенное действие MUST иметь глобально уникальное постоянное
имя и MAY использовать любой разрешённый `CALL`. Система SHALL считать
ближайший enclosing `CALL SUBAGENT` исполнителем всех вызовов в этом блоке и
MUST NOT создавать альтернативную inline-форму payload.

#### Scenario: Субагент вызывает один skill
- **WHEN** описание поручает субагенту выполнить один доступный skill
- **THEN** документ помещает именованное действие `CALL SKILL` во вложенный блок `CALL SUBAGENT`

#### Scenario: Субагент выполняет последовательность
- **WHEN** описание поручает субагенту несколько последовательных действий
- **THEN** документ создаёт под `CALL SUBAGENT` вложенный нумерованный список с постоянным именем каждого действия

### Requirement: Управляющие связи полны и ограничены
Каждое решение SHALL объявлять конечный набор вариантов и продолжение каждого
варианта. Каждое повторение SHALL иметь постоянное имя, условие завершения,
положительный предел попыток и явный исход при исчерпании предела. Произвольные
переходы и неограниченные повторения MUST NOT создаваться.

#### Scenario: Вариант решения не имеет продолжения
- **WHEN** один из объявленных вариантов не ведёт к действию, завершению или обращению к человеку
- **THEN** статическая проверка отклоняет flow как несвязный

#### Scenario: Для возврата не указан предел
- **WHEN** описание требует возвращаться к предыдущим действиям, но не задаёт положительный предел попыток
- **THEN** составитель спрашивает пользователя о пределе до записи flow

### Requirement: Параллельная работа выражается только явно
Составитель MUST создавать `PARALLEL` только когда пользователь явно просит
параллельную работу или однозначно описывает одновременные независимые действия.
Каждое дочернее действие SHALL иметь постоянное имя. Если намерение выполнять
действия параллельно неоднозначно, составитель MUST оставить их
последовательными либо запросить уточнение.

#### Scenario: Пользователь просит две независимые проверки одновременно
- **WHEN** описание явно требует параллельно выполнить две независимые проверки
- **THEN** составитель создаёт один видимый `PARALLEL`-блок с двумя дочерними действиями

#### Scenario: Параллельность не указана
- **WHEN** описание перечисляет две проверки, но не определяет порядок их выполнения
- **THEN** составитель оставляет проверки последовательными либо уточняет намерение пользователя

### Requirement: Составитель выполняет лёгкую статическую проверку
До возврата результата `usw-create-flow` SHALL проверить обязательные разделы,
уникальность постоянных имён, существование внутренних ссылок, полноту путей
решений, положительные пределы повторений и разрешимость указанных skills,
scripts и nested flows. Система MUST исправить обнаруженную
ошибку или запросить недостающую информацию до успешного завершения.

Проверка MUST NOT запускать действия, доказывать фактическую исполнимость веток
или утверждать реальную независимость будущего параллельного исполнения.

#### Scenario: Статическая проверка прошла
- **WHEN** все проверяемые свойства созданного документа подтверждены
- **THEN** система возвращает имя, расположение, версию и действия flow без исполнения

#### Scenario: Ссылка не разрешается
- **WHEN** действие ссылается на отсутствующий skill, небезопасный script или отсутствующий nested flow
- **THEN** система не сообщает об успешном создании, пока ссылка не исправлена

#### Scenario: Runtime-свойство нельзя доказать
- **WHEN** структура документа валидна, но фактический результат будущего условия неизвестен
- **THEN** проверка не исполняет условие и не выдаёт предположение за подтверждённый runtime-факт

### Requirement: Новая версия 1 не дублирует write-contract
Новый flow версии `1` SHALL содержать секции `Контракт` и `Порядок действий`,
но MUST NOT содержать декларации `Пишет` или раздел `Полномочия записи`.
Линейный runner SHALL принимать эту concise-форму и существующую verbose-форму
версии `1`. Для concise skill-step runner SHALL использовать write-contract
разрешённого executor. Запуск script SHALL по-прежнему требовать отдельного
разрешения на точный project-local путь.

Verbose-flow MUST сохранять прежнюю проверку: `Пишет` каждого шага входит в
общие полномочия и совпадает с capability contract skill. Смешанная форма, где
присутствует только часть legacy write-metadata, MUST быть отклонена.

#### Scenario: Создан новый линейный flow
- **WHEN** пользователь выбирает версию `1`
- **THEN** составитель записывает линейные skill/script шаги без `Пишет` и раздела полномочий записи

#### Scenario: Concise-flow вызывает skill
- **WHEN** runner разрешает skill-step нового concise-flow
- **THEN** его write-contract берётся из зарегистрированного executor до первого вызова

#### Scenario: Запускается существующий verbose-flow
- **WHEN** version-1 flow содержит полный прежний write-contract
- **THEN** runner проверяет и исполняет его без миграции

#### Scenario: Write-metadata указана частично
- **WHEN** flow содержит `Пишет` без раздела полномочий либо раздел без `Пишет` каждого шага
- **THEN** parser отклоняет неоднозначный контракт до исполнения

### Requirement: Обе версии сохраняют свои границы
Версия `1` SHALL оставаться строгой линейной последовательностью skills и
project-local scripts и MUST NOT получать structured-конструкции. Новая
concise-форма MUST NOT требовать миграции существующих verbose-flow.

#### Scenario: Создаётся версия 1
- **WHEN** пользователь после вопроса выбирает версию `1`
- **THEN** `usw-create-flow` использует concise линейный контракт без structured-маркеров и write-metadata

#### Scenario: Существующий flow версии 1 запускается после изменения
- **WHEN** пользователь запускает ранее созданный валидный flow версии `1`
- **THEN** существующие разбор и исполнение работают без миграции документа

### Requirement: Детали выбранной версии раскрываются по требованию
`usw-create-flow` SHALL держать выбор версии и общие safety-инварианты в
основном skill. После выбора версии система MUST загрузить ровно один прямой
reference с формой, проверкой и отчётом выбранной версии и MUST NOT загружать
оба version-specific reference до выбора.

#### Scenario: Версия ещё не выбрана
- **WHEN** structured-флаг отсутствует и пользователь ещё не ответил на вопрос о версии
- **THEN** система не загружает ни линейный, ни structured reference

#### Scenario: Выбрана линейная версия
- **WHEN** пользователь выбирает версию `1`
- **THEN** система загружает только reference версии `1`

#### Scenario: Выбрана structured-версия
- **WHEN** передан structured-флаг или пользователь выбирает `version-2`
- **THEN** система загружает только reference `version-2`
